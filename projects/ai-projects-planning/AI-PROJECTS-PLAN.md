## 1. 背景与设计思路

### 1.1 当前问题诊断

经过对 python4office.cn 项目文件的全面盘点，发现以下核心问题：

| 问题 | 具体表现 | 影响 |
|------|----------|------|
| **多线作战，没有聚焦** | 同时推进：录课、公众号、口播稿、线下活动、量化交易、网站运营，共6条线 | 每条线都停留在"维持"状态，没有一条进入"领先"状态 |
| **内容平行，没有递进** | 747篇广告文、135篇口播稿、717篇课程内容，三者之间没有转化路径 | 读者看完一篇文章就走了，没有"下一步" |
| **形式不固定，人设模糊** | 同时做中长视频、短视频、文章、卖课，没有哪个形式做得很好 | 用户记不住"晚枫是谁"，只记得"有个教Python的博主" |
| **产品与内容脱节** | python-office（下载量40万+）和课程内容有断层，没有深度串联 | 开源项目的信任没有转化为课程销售的信任 |

### 1.2 核心决策

基于以上诊断，做出以下战略决策：

**决策1：聚焦"AI+一人公司（OPC）"方向**
- 理由：月入2万+的自媒体变现是最好的"一人公司"案例；python-office 是真实的产品资产；市场正在爆发（2024年国内OPC市场规模已突破800亿元）
- 参考资料：北京大学《2026年Agentic Coding》、清华大学《2026一人公司（OPC）发展研究报告》

**决策2：固定"公众号文章"为主形式**
- 理由：可控性高（自有平台）、制作成本低（已有747篇经验）、变现直接（文末直接卖课）、可积累性强（SEO+长期搜索）
- 其他形式重新定位：短视频降级为"公众号文章的衍生品"；中长视频降级为"每月1次开发实录"；课程不是"形式"，是"变现产品"

**决策3：给所有AI热点加固定叙事角度**
- 不是"报道AI热点"，而是"晚枫在用这个做一人公司"
- 改开头：从"XX发生了"改为"我用XX做了YY，今天算笔账"
- 改结尾：从"相关阅读"改为"下一篇写我用AI把XX效率提高N倍"

### 1.3 设计思路

**"用AI开发真实软件"作为核心叙事**

通过公开记录用AI开发软件的完整过程（Build in Public），建立"晚枫是用AI做真实产品的实战者"人设，而非"AI资讯号"。

**内容分层体系**

```
碎片化思考（X平台短帖）
    ↓
深度长文（公众号）—— 主形式
    ↓
开发日记（系列文章）—— 建立期待
    ↓
软件产品（可下载/可购买）—— 变现载体
    ↓
社群/课程（晚枫OPC实战社群）—— 持续服务
```

**变现三层体系**

| 层级 | 产品 | 定价 | 收入测算（理论值） |
|------|------|------|---------------------|
| 第一层 | OPC入门课（python-office+AI基础） | 99-199元 | 月收入7.98万（40万粉×5%打开×1%转化×399元） |
| 第二层 | OPC进阶课（完整搭建工作流） | 399-699元 | 同上 |
| 第三层 | OPC训练营（直播+1对1+结果交付） | 999-2999元 | 年收入48万（1万活跃粉×1%转化×399元） |

## 2. 四个AI项目开发计划

### 项目1：AI语音输入

#### 2.1.1 项目定位

**一句话描述**：基于 Whisper 的本地离线语音转文字工具，面向需要高频文字输入的职场人（程序员、作家、公众号编辑）。

**核心价值**：
- 隐私安全：100%离线运行，数据不上传
- 成本低：开源模型，无订阅费
- 中文优化：针对中文口语场景优化

**与目标用户的关系**：
- 公众号编辑：可以用语音写稿，提高3-5倍写作速度
- 程序员：可以用语音写代码注释、commit message
- 一人公司创始人：所有文字工作都可以语音输入

#### 2.1.2 技术方案

| 技术层 | 技术方案 | 理由 |
|--------|----------|------|
| **语音识别引擎** | OpenAI Whisper（本地部署） | 开源、免费、中文支持好；参考项目：WhisprRT（https://github.com/zhengjim/WhisprRT） |
| **前端界面** | Electron 或 Tauri（跨平台桌面应用） | 轻量、跨平台（Windows/Mac/Linux） |
| **后端服务** | Python FastAPI（参考 WhisprRT 架构） | 轻量、异步、易部署 |
| **音频处理** | FFmpeg（Whisper 依赖） | 行业标准，支持所有音频格式 |
| **模型大小选择** | 提供多个版本：tiny（75MB）、base（140MB）、small（460MB） | 让用户根据设备性能选择 |

#### 2.1.3 开发阶段与任务分解

**阶段1：MVP（最小可行产品）—— 2周**

| 任务 | 具体内容 | 里程碑 |
|------|----------|---------|
| 环境搭建 | 安装 Whisper、FFmpeg、创建 Python 项目 | 能运行 Whisper 命令行转写音频文件 |
| 核心功能1 | 实现"选择音频文件 → 转文字 → 显示结果" | 完成基础转写功能 |
| 核心功能2 | 实现"实时录音 → 实时转写" | 完成实时转写功能 |
| 简单UI | 用 Gradio 或 Tkinter 做简单界面 | 有图形界面，不依赖命令行 |
| 测试 | 测试中文识别准确率、实时延迟 | 中文准确率>85%，延迟<2秒 |

**阶段2：产品化 —— 2周**

| 任务 | 具体内容 | 里程碑 |
|------|----------|---------|
| 打包分发 | 用 PyInstaller 打包成可执行文件 | Windows/Mac 用户可以直接下载使用 |
| 快捷键激活 | 全局快捷键（如 Ctrl+Alt+V）激活语音输入 | 不切换窗口就能用语音输入 |
| 导出功能 | 支持导出 TXT/MD/直接复制到剪贴板 | 转写结果可以直接用 |
| 错误处理 | 添加录音失败、模型加载失败的友好提示 | 用户体验基本可用 |

**阶段3：AI增强 —— 3周**

| 任务 | 具体内容 | 里程碑 |
|------|----------|---------|
| 标点智能添加 | 用 AI 模型优化转写结果的标点 | 转写结果可以直接用，不需要手动加标点 |
| 口语清理 | 自动删除"嗯、啊、那个"等口语词 | 输出更干净 |
| 专业术语优化 | 针对编程、法律等垂直领域优化识别 | 编程术语识别准确率>95% |
| 多语言支持 | 支持中英文混说 | 中英文混说场景可用 |

**阶段4：发布与迭代 —— 持续**

| 任务 | 具体内容 | 里程碑 |
|------|----------|---------|
| GitHub 开源 | 发布到 GitHub，写完整 README | 获得第一批用户和反馈 |
| python-office 集成 | 添加 `office.voice_input()` 一行代码调用 | 与现有用户群体打通 |
| 持续迭代 | 根据用户反馈修复 bug、添加功能 | 持续维护 |

#### 2.1.4 优先级排序

| 优先级 | 功能 | 理由 |
|--------|------|------|
| P0 | 音频文件转文字 | 核心功能，必须先有 |
| P0 | 简单UI | 没有UI普通用户不会用 |
| P1 | 实时录音转写 | 核心功能，但比文件转写复杂 |
| P1 | 打包分发 | 有了功能必须能让用户用到 |
| P2 | 快捷键激活 | 提高用户体验 |
| P2 | AI增强（标点、口语清理） | 差异化功能，但MVP阶段可以没有 |
| P3 | 多语言支持 | 进阶功能 |

#### 2.1.5 里程碑目标

| 时间 | 里程碑 | 可衡量目标 |
|------|--------|-------------|
| 第2周 | MVP完成 | 能转写中文音频，准确率>85% |
| 第4周 | 产品化完成 | 有可执行文件，100个内测用户 |
| 第7周 | AI增强完成 | 标点准确率>95%，口语清理可用 |
| 第8周 | 正式发布 | GitHub 发布，目标100 Star |

#### 2.1.6 参考来源

- **WhisprRT**（https://github.com/zhengjim/WhisprRT）：基于 Whisper 的本地实时语音转文字工具，架构参考
- **Whisper 官方**（https://github.com/openai/whisper）：模型原理和用法
- **Whisper Web**（https://github.com/xenova/whisper-web）：浏览器端运行 Whisper 的参考实现

### 项目2：AI发票识别

#### 2.2.1 项目定位

**一句话描述**：1行Python代码批量识别增值税发票并导出Excel，面向需要频繁报销的职场人和财务人员。

**核心价值**：
- 效率高：1分钟处理100+张发票（已有 poocr 项目验证）
- 成本低：开源方案，无按张收费
- 准确率高：基于深度学习 OCR，准确率>95%

**与目标用户的关系**：
- 职场人：每月报销不再需要手动录入发票信息
- 财务人员：批量处理公司员工的发票
- 一人公司创始人：自动整理经营成本，报税更轻松

#### 2.2.2 现有基础

**重要发现**：本项目已有开源实现 **poocr**（程序员晚枫开发），发表于2025年11月（参考来源：https://cloud.tencent.com/developer/article/2583707）。

**现有功能**：
- 1行代码识别增值税发票
- 批量处理
- 导出Excel

**待优化方向**（基于竞品分析）：
- 竞品 Invoice（https://zhuanlan.zhihu.com/p/696661855）提供了基于 Flask 的微服务框架，可以借鉴
- 竞品 myocr2-invoice（https://blog.csdn.net/weixin_45219221/article/details/147555043）使用 RT-DETRv2 目标检测技术，识别准确率更高

#### 2.2.3 技术方案

| 技术层 | 技术方案 | 理由 |
|--------|----------|------|
| **OCR引擎** | PaddleOCR 或 RT-DETRv2（目标检测）+ CRNN（文字识别） | 中文发票识别准确率最高 |
| **后端服务** | Flask（参考 Invoice 项目架构） | 轻量、易部署、适合微服务 |
| **前端界面** | 简单的 Web UI（Flask 模板） | 降低用户使用门槛 |
| **导出格式** | Excel（.xlsx） | 财务标准格式 |
| **批量处理** | 多线程/异步处理 | 提高批量处理速度 |

**阶段1：优化现有 poocr 项目 —— 1周**

### 项目3：AI公众号编辑器

#### 2.3.1 项目定位

**一句话描述**：AI 辅助的微信公众号 Markdown 编辑器，支持实时预览、AI 智能排版、公众号格式一键转换。

**核心价值**：
- 效率高：Markdown 写作 + 实时预览，写作速度提高2倍
- 排版美：AI 自动排版，不需要手动调整样式
- 格式兼容：一键转换公众号格式，直接复制粘贴

**与目标用户的关系**：
- 公众号编辑：提高写作和排版效率
- 技术博主：代码块、公式渲染效果好
- 一人公司创始人：内容生产是核心工作，效率至关重要

#### 2.3.2 竞品分析

| 竞品 | 优势 | 不足 | 我们的差异化 |
|------|------|------|-------------|
| **doocs/md**（https://blog.csdn.net/2401_87629362/article/details/161901144） | 开源、实时预览、AI辅助写作 | 需要本地部署，普通用户有门槛 | 提供在线版本，开箱即用 |
| **SnapWrite**（https://blog.csdn.net/weixin_42429220/article/details/156687197） | AI智能排版、语义理解 | 闭源，无法定制 | 开源，可以定制 |
| **微信公众号助手**（https://zhuanlan.zhihu.com/p/22004204231） | 一键导入、格式完美保留 | 无AI辅助 | 加入AI辅助写作 |

#### 2.3.3 技术方案

| 技术层 | 技术方案 | 理由 |
|--------|----------|------|
| **编辑器内核** | CodeMirror 6 或 Monaco Editor（VS Code 同款） | 功能强大、扩展性高 |
| **Markdown 解析** | marked.js 或 markdown-it | 轻量、插件生态好 |
| **实时预览** | 分屏预览（左写右看） | 用户体验好 |
| **AI 辅助** | 接入 OpenAI API 或国内大模型 API（DeepSeek/通义千问） | 功能强大、成本低 |
| **公众号格式转换** | 自定义 CSS + HTML 生成逻辑 | 公众号有特殊格式要求 |
| **部署** | 静态网站（Vercel 或 GitHub Pages） | 无需服务器，成本低 |

### 4.2 技术方案选择理由

#### 4.2.1 为什么语音输入选 Whisper？

| 对比 | Whisper | 其他方案（如百度语音、讯飞语音） |
|------|---------|---------------------------|
| 成本 | 开源免费 | 按调用次数收费，长期使用成本高 |
| 隐私 | 完全离线 | 需要上传到云端，隐私有风险 |
| 中文支持 | 好（训练数据包含中文） | 好 |
| 可定制性 | 高（开源，可以微调） | 低（闭源API） |

**结论**：Whisper 最符合"一人公司"的低成本、高隐私需求。

---

#### 4.2.2 为什么发票识别选 PaddleOCR + RT-DETRv2？

| 对比 | PaddleOCR + RT-DETRv2 | 其他方案（如Tesseract） |
|------|------------------------|---------------------------|
| 中文准确率 | 高（针对中文优化） | 低（通用OCR，中文效果差） |
| 发票场景 | 好（有专门针对发票的模型） | 差（需要自己训练） |
| 开源 | 是 | 是 |

**结论**：PaddleOCR + RT-DETRv2 是中文发票识别的最佳开源方案。

---

#### 4.2.3 为什么公众号编辑器选 CodeMirror 6 + Next.js？

| 对比 | CodeMirror 6 | 其他方案（如Monaco Editor） |
|------|---------------|---------------------------|
| 体积 | 小（按需加载） | 大（VS Code同款，功能多但体积大） |
| 扩展性 | 高（插件生态好） | 高 |
| 学习成本 | 中 | 高 |

**结论**：CodeMirror 6 更适合轻量级编辑器场景。

---

#### 4.2.4 为什么个人网站继续用 Hexo，但逐步迁移到 Next.js？

| 对比 | Hexo | Next.js |
|------|-------|---------|
| 现有内容 | 已有747篇+文章，迁移成本高 | 需要从头迁移 |
| 动态功能 | 不支持（纯静态） | 支持（API路由、SSR） |
| AI生成内容 | 不方便 | 方便（可以直接调用API生成内容） |

**结论**：短期内继续用 Hexo（保护现有内容资产），但新功能（如AI聊天助手）用 Next.js 实现，逐步迁移。

### 4.3 功能优先级安排理由

#### 4.3.1 为什么"核心功能"优先级最高？

**理由**：没有核心功能，其他功能都没有意义。必须先让用户"能用"，再让用户"好用"。

**具体应用**：
- 语音输入：必须先能"转文字"，再优化"标点、口语清理"
- 发票识别：必须先能"识别"，再优化"智能纠错、自动分类"
- 公众号编辑器：必须先能"写Markdown + 实时预览"，再添加"AI辅助"
- 个人网站：必须先有"AI摘要 + 标签优化"，再添加"AI聊天助手"

---

#### 4.3.2 为什么"降低使用门槛"优先级高于"高级功能"？

**理由**：一人公司创始人时间有限，如果一个工具需要花1小时学习，他们可能不会用。必须做到"开箱即用"。

**具体应用**：
- 语音输入：先提供"可执行文件（.exe / .dmg）"，而不是"需要命令行运行Python脚本"
- 发票识别：先提供"Web UI（拖拽上传）"，而不是"需要写Python代码调用"
- 公众号编辑器：先提供"在线版本"，而不是"需要本地部署"
