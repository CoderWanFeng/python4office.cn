## 🧐 一、Codex for OSS 到底是什么？

先说结论：**这是一项 OpenAI 官方推出的开源支持计划**，英文全称是 **Codex for Open Source**。

简单讲：OpenAI 觉得，开源项目的维护者工作量太大、太苦逼了，于是拿出一部分资源，免费送给这些人用。

入选之后，你会拿到三样东西：

| 福利 | 数量 | 用途 |
|------|------|------|
| **ChatGPT Pro（含 Codex）** | 6 个月 | 日常写代码、看 PR、写文档、回 Issue |
| **API 额度** | 按项目情况 | 接 PR 审查、自动化、版本发布工作流 |
| **Codex Security** | 有条件 | 给关键项目做深度安全扫描 |

**这三项加起来，市场价超过 1 万人民币。**

也就是说，只要你符合条件，OpenAI 直接给你"充钱"——而且是 6 个月。

### 它的官方介绍页在哪？

- 中文版入口：**https://openai.com/zh-Hans-CN/form/codex-for-oss/**
- 英文版入口：https://openai.com/form/codex-for-oss/

## ✅ 二、谁能申请？

![](https://raw.atomgit.com/user-images/assets/5027920/63619f69-3f14-4627-8002-4c3a190a669f/image.png 'image.png')

这是最关键的一步——**很多人不知道自己其实符合条件**。

官方写得比较抽象，我翻译成大白话：

### 1. 你必须是「真正的维护者」，不是围观群众

具体来说：

- 你是某个开源项目的 **主要作者 / 核心维护者**
- 你经常要 **回 Issue、看 PR、发版本**
- 你对仓库有 **写权限**（能直接 push 代码的那种）

如果你只是给别人的项目点过 Star、提过几次 PR，那大概率不行。

### 2. 你的项目得有「真实使用量」

官方明确说了：

> 我们优先考量具有实际使用量、广泛采用，或对软件生态系具明确重要性的项目。

翻译一下：

- 你的项目有人用（下载量、Star 数、被引用）
- 你的项目在某个生态里有分量（哪怕不是超大项目）
- 你的项目最近还在持续维护（最近 90 天有提交）

**注意**：官方 **没有硬性 Star 门槛**，没有说"必须 1 万 Star 才能申请"。

所以哪怕你的项目只有几百 Star，但解决了一个真实痛点，也可以试试。

### 3. 你能用清楚的语言说清楚两件事

- **为什么你的项目值得支持**（量化数据 + 真实场景）
- **你打算怎么用 Codex**（具体工作流，不是"我想用 AI"）

只要你能说清楚这两点，**就已经超过 60% 的申请者**。

## 📋 三、申请前，你要准备好这些材料

填表之前，先把这些东西准备好，**否则你会反复退出页面，很烦躁**。

![](https://raw.atomgit.com/user-images/assets/5027920/0b99506d-6c17-4cd3-9326-f898861bb55f/image.png 'image.png')

### 1. 一个能正常登录的 ChatGPT 账号

这一点是官方硬性要求，没账号的先去注册。

### 2. 你的 GitHub 个人主页链接（必须公开）

很多小伙伴不知道：**GitHub 个人主页默认就是公开的**，但你最好确认一下。

设置路径：GitHub → 右上角头像 → Settings → Profile → 勾选 **Public profile**。

### 3. 你要申请的那个开源项目的资料

提前整理好这些信息：

| 项目 | 内容 |
|------|------|
| **仓库地址** | 比如 https://github.com/xxx/xxx |
| **你的角色** | 作者 / 核心维护者 / 组织管理员 |
| **最近 90 天的活跃度** | 提交数、PR 处理数、Issue 回复数 |
| **影响力数据** | Star 数、下载量、依赖它的项目 |
| **生态重要性** | 它在某个领域、某个工作流里扮演什么角色 |

> 💡 **小技巧**：提前把这些内容写在一个文本文件里，填表时直接复制粘贴。

### 4. 你打算怎么用 Codex

这是表格里最关键的两道主观题，需要 **提前写好草稿**。

- [Codex 入门指南，从零基础到实战，看这一篇就够了！](https://mp.weixin.qq.com/s/ilvNENMEiPy2uEYyDflvQA)

我给你一个万能结构：

```
【项目为什么值得支持】
我的项目 XXX，主要解决 XXX 问题。
目前有 XXX 个 Star、XXX 月下载量，被 XXX 项目依赖。
最近 90 天处理了 XXX 个 PR、回覆了 XXX 个 Issue。

【我打算怎么用 Codex】
1. PR 首轮审查：自动检查代码风格、安全问题
2. Issue 自动分类：根据标签自动分流
3. 版本说明生成：根据提交记录自动写 release notes
4. 文档辅助：自动生成使用示例
```

"**写具体，不要写"我想体验一下"这种话。**"

## 🚀 四、一步一步教你填表

下面进入实操环节，**我截图式地教你每一行怎么填**。

### 第 1 步：打开申请入口

浏览器访问：**https://openai.com/zh-Hans-CN/form/codex-for-oss/**

点击页面上的 **「Apply today」** 按钮，进入申请表。

### 第 2 步：填写基础信息

这一部分比较简单：

- 你的姓名
- 你的邮箱（建议用 GitHub 注册邮箱）
- 你所在的国家/地区
- 你的 ChatGPT 账号邮箱（必须能登录）

> ⚠️ 两个邮箱最好用一个，**否则容易出现账号对不上的问题**。

### 第 3 步：填写项目信息

这一部分是表格核心，要认真填。

**3.1 仓库地址**
直接粘贴你的 GitHub 仓库完整 URL。

**3.2 你的维护者角色**
- 不要写"用户"或"粉丝"
- 要写"主要作者"、"核心维护者"、"具备写权限的协作者"

**3.3 为什么这个仓库值得支持？** ⭐⭐⭐⭐⭐ 最重要
- 写数据，不要写形容词
- 写下游依赖，不要写"我觉得挺重要"

**错误示范**：
> "这是一个很好的项目，解决了大家日常开发中的痛点。"

**正确示范**：
> "本项目是一个 Python 办公自动化库，目前 GitHub 1.2 万 Star，PyPI 月下载量 5 万+，被 XX、XX 等知名项目依赖。最近 90 天合并了 87 个 PR、回复了 320 个 Issue、发布了 4 个版本。"

**3.4 你打算怎么用 Codex / API 额度？** ⭐⭐⭐⭐⭐ 最重要
- 写具体工作流
- 写自动化场景
- 写你能"接手审查 AI 输出"的机制

**错误示范**：
> "我打算用 Codex 帮我写代码、查 Bug。"

**正确示范**：
> "1) PR 首轮审查：Codex 自动检查代码规范、潜在安全问题，生成审查意见草稿，我人工复核后再回复作者；2) Issue 自动分流：基于 Issue 内容打标签、归类到不同模块的负责人；3) Release Notes 自动生成：根据 commits 自动写版本说明草稿；4) 我会用 GitHub Actions 把 Codex 接到 CI 流程，所有 AI 输出都会人工复核后再合并。"

### 第 4 步：选择你感兴趣的福利

这三项可以多选：

| 福利 | 适合谁 |
|------|------|
| ChatGPT Pro 6 个月 | 所有维护者都建议勾选 |
| Codex Security | 做底层框架、安全相关项目的勾选 |
| API 额度 | 想要自动化工作流的勾选 |

**建议全部勾选**，反正不勾白不勾。

### 第 5 步：提交申请

填完之后，点 **「Submit」** 提交。

官方说：「我们会持续审核收到的申请」，**没有承诺审核周期**，但根据经验，通常 1-4 周会有结果。

结果会通过你填写的邮箱通知，所以 **邮箱一定要填对**。

## 💡 五、提高通过率的 3 条"潜规则"

最后，分享 3 条不会写在官网上的小技巧：

### 1. 项目页面要"看起来活跃"

如果你的 GitHub 项目页面半年没更新，**通过率会大打折扣**。

提交申请前，花 1 小时：

- 更新 README
- 把积压的 Issue 回一轮
- 合几个小 PR
- 发一个版本

不是让你造假，是 **让项目看起来还活着**。

### 2. 不要用 AI 直接生成申请文案

是的，**OpenAI 是这个计划的发起方**，他们一眼就能看出你是不是用 ChatGPT 写的。

最好的写法：**你自己写骨架，让 AI 帮你润色**。

### 3. 提名别人也可以

官方明确说了：「**开发者可以为自己的项目申请，也可以提名其他维护者**」。

所以如果你身边有靠谱的开源大佬，**去提名他**，你和他都有机会。
